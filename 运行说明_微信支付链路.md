# Linux 微信支付链路运行说明

## 目标

本文档用于记录当前 `linux_wechat_pay` 与 Message Bot 统一结算系统的实际运行方式、关键日志、已完成优化和日常操作方法，便于后续维护与排障。

## 当前实际链路

当前支付确认主链路如下：

1. 用户在 Message Bot 内创建 `linux_wechat` 订单。
2. Bot 向用户发送收款码并进入等待支付状态。
3. Linux 侧监听微信数据库变化。
4. 优先从 `biz_message_0.db` 提前识别支付到账消息。
5. 将支付结果回调到 Message Bot 的 `/api/payment/notify/linux_wechat`。
6. Message Bot 按金额唯一匹配待支付订单并完成入账。
7. `session.db` 仍保留为后备检测来源，但已做跨来源去重，避免重复通知。

## 当前检测策略

### 主来源

`biz_message_db`

说明：

1. 这是当前优先使用的较早支付信号源。
2. 成功时可直接解析到付款人名称，例如 `如愿以偿`。
3. 日志特征为：`[BIZ DB DETECTED]`

### 后备来源

`session_db`

说明：

1. 当 `biz_message_db` 未提前命中时，仍可通过 `session.db` 识别到账摘要。
2. 日志特征为：`[DB DETECTED]`
3. 已与 `biz_message_db` 做统一去重，正常情况下不会再次产生有效重复回调。

### 已停用来源

`ocr`

说明：

1. OCR 不再参与当前支付确认流程。
2. `start_with_display.sh` 已跳过 OCR 自检。
3. 主程序也不再启动 OCR 持续扫描线程。

## 已完成优化

本次已完成的稳定性与性能优化如下：

1. 重复解密去重：`db_monitor` 与 `message_monitor` 不再同时重复执行 `decrypt_db.py`。
2. 解密刷新后立即重扫：避免解密完成后继续空等。
3. `decrypt_db.py` 增量化：未变化数据库直接跳过。
4. 解密结果原子替换：降低读取半成品数据库导致的损坏错误。
5. `message_monitor` 对临时 SQLite 异常做容错处理。
6. 新增 `biz_message_db` 早期支付检测。
7. 启动时建立 `biz_message` 基线，避免旧支付历史重放。
8. `biz_message_db` 与 `session_db` 已做跨来源去重。
9. 回调顶层 `source` 已透传真实检测来源。
10. 启动脚本已跳过 OCR 自检。

## 启动后正常日志特征

### 启动阶段

应能看到类似日志：

```text
Skipping OCR runtime self-check (OCR disabled in current payment flow).
Biz message baseline established at timestamp=...
Starting session DB monitor...
Starting message DB monitor...
System started successfully!
```

### 支付命中阶段

优先命中时通常会看到：

```text
[BIZ DB DETECTED] 2.0元 from 如愿以偿 ...
Callback sent successfully to http://192.168.100.8:5000/api/payment/notify/linux_wechat
```

如后备命中，则可能看到：

```text
[DB DETECTED] 2.0元 from 未知 ...
```

## 成功判定标准

满足以下条件即可认为链路正常：

1. 一笔订单只产生一次有效支付通知。
2. Message Bot 能正确匹配到待支付订单。
3. 不出现 `ORPHAN_...` 孤儿支付记录。
4. 用户最终能收到“充值成功”消息。

## 常见问题

### 1. 出现 `No matching order found for payment`

说明：

1. 这是 `linux_wechat_pay` 内部订单管理器未匹配到本地订单。
2. 当前实际业务匹配主要由 Message Bot 统一结算系统按金额唯一匹配完成。

影响：

1. 只要 Message Bot 正常完成支付匹配，这条日志不影响最终到账。

### 2. 出现 `database disk image is malformed`

说明：

1. 通常是解密输出文件处于切换或半写入瞬间。
2. 当前已通过原子写入和容错处理降低影响。

### 3. 出现重复支付通知

当前已通过以下方式抑制：

1. 启动时建立 `biz_message` 基线。
2. `biz_message_db` 与 `session_db` 做统一时间窗口去重。

若后续仍出现重复，需要检查最新日志中是否同时出现：

1. `[BIZ DB DETECTED]`
2. `[DB DETECTED]`
3. `ORPHAN_...`

## 当前结论

截至当前版本，这条 Linux 微信支付回调链路已经满足以下目标：

1. 可稳定运行。
2. 已与 Message Bot 统一结算系统打通。
3. 付款人名称可在 `biz_message_db` 场景下正确解析。
4. OCR 已退出实际支付确认链路。
5. 重复回调与孤儿单问题已压制到可接受状态。

如后续继续优化，优先方向应是：

1. 进一步弱化 `message_monitor` 在支付场景中的参与度。
2. 视需要降低 `order_manager` 未匹配日志噪音。
3. 若要更强绑定，可考虑把 Message Bot 订单号与 Linux 侧待支付信息做映射透传。
